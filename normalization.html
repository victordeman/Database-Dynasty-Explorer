Database Normalization

We now introduce normal forms: Techniques to eliminate redundancy and ensure data integrity in relational databases.

What is Normalization?

Normalization is a systematic process of organizing data in a relational database to minimize redundancy and dependency by dividing large tables into smaller, related tables while maintaining data integrity. Introduced by E.F. Codd, it follows a series of normal forms (NF) that progressively enforce stricter rules.

Benefits: Reduces anomalies during insert/update/delete operations, saves storage, and improves query performance. However, over-normalization can lead to complex joins.

Why Normalize?

  • Eliminate data redundancy
  • Prevent update anomalies
  • Ensure referential integrity

Normal Forms Hierarchy

1NF → 2NF → 3NF → BCNF → 4NF → 5NF (Higher forms address specific anomalies)

First Normal Form (1NF)

Atomic values: Each cell contains a single value, no repeating groups, and a primary key.

Unnormalized Example

Student IDNameCourses
101AliceMath, CS101

Issue: Multi-valued attribute in "Courses".

1NF Normalized

Student IDNameCourse
101AliceMath
101AliceCS101

Solution: Separate rows for each course; add primary key (composite if needed).

-- SQL for 1NF
CREATE TABLE StudentCourses (
    StudentID INT,
    Name VARCHAR(50),
    Course VARCHAR(50),
    PRIMARY KEY (StudentID, Course)
);

Second Normal Form (2NF)

1NF + No partial dependencies: All non-key attributes fully depend on the entire primary key (eliminates partial dependency in composite keys).

Example Issue

OrderIDProductIDProductNameQuantity
1101Laptop2

Partial dependency: ProductName depends only on ProductID, not full key (OrderID, ProductID).

2NF Solution

Split into Orders and Products tables.

Products(ProductID, ProductName)
Orders(OrderID, ProductID, Quantity)

Third Normal Form (3NF)

2NF + No transitive dependencies: Non-key attributes depend only on the primary key, not other non-key attributes.

Example Issue

StudentIDNameAdvisorAdvisorOffice
101AliceDr. SmithRoom 101

Transitive: AdvisorOffice depends on Advisor, not directly on StudentID.

3NF Solution

Split into Students and Advisors tables.

Advisors(Advisor, Office)
Students(StudentID, Name, Advisor)

Higher Normal Forms (BCNF, 4NF, 5NF)

Beyond 3NF for advanced anomalies:

BCNF

Every determinant is a candidate key (handles overlapping keys).

4NF

No multi-valued dependencies (e.g., separate unrelated multi-value attributes).

5NF

No join dependencies (full decomposition without loss).

-- Example: Decomposing to BCNF
-- Original: Teaches(Teacher, Subject, Student)
-- BCNF: TeacherSubject(Teacher, Subject), SubjectStudent(Subject, Student)

In practice, 3NF is often sufficient; higher forms for complex schemas.

Key Takeaways

Data Integrity

Normal forms prevent anomalies by structuring dependencies properly.

Progressive Refinement

Start with 1NF, aim for 3NF; balance with performance needs.

Practical Application

Essential for schema design in RDBMS like PostgreSQL and MySQL.

"Normalization is the cornerstone of efficient, anomaly-free databases."
